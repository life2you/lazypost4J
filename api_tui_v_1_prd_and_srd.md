# API TUI 工具 V1 产品文档（PRD）+ 需求文档（SRD）

## 1. 文档信息

- 产品名称：API TUI（本地接口扫描与调试工具）
- 版本：V1.0
- 文档类型：产品文档 + 需求规格说明
- 目标版本范围：仅支持 Spring MVC / Spring Boot 项目
- 支持操作系统：macOS、Linux
- 产品形态：本地命令行 + TUI 工具
- 核心能力：给定 Java 项目目录，扫描本地源码中的 HTTP 接口，并在 TUI 中完成浏览、检索、参数编辑与请求发送
- **实现技术栈（V1 基线）**：**Rust**（终端 UI 与扫描、HTTP 客户端均在本进程内；**用户运行本工具不依赖本机安装 Java/JVM**）
- **TUI 体验目标**：全屏、多面板、键盘优先，交互与信息密度参考 **yazi / lazygit** 等现代终端工具（非简单菜单式 CLI）

# 第一部分：产品文档（PRD）

## 2. 背景与问题定义

在 Java 后端项目中，接口调试通常依赖以下方式：

1. 人工阅读 Controller 源码或注解
2. 借助 Swagger / OpenAPI 页面，但很多项目未维护或信息不完整
3. 手工在 Postman、curl、IDE HTTP Client 中拼接请求
4. 在多模块项目中反复定位接口定义、参数和环境配置

当前问题主要有：

- 接口分散在项目源码中，查找成本高
- 文档与代码容易不一致
- 没有 Swagger 的项目调试门槛高
- 从“找到接口”到“发起请求”之间步骤过多
- 对终端工作流用户不友好，无法形成可复用的本地 API 工作台

因此需要一个面向本地开发者的 TUI 工具：

- 输入一个 Java 项目目录
- 自动扫描 Spring MVC / Boot 接口
- 生成统一的本地 API 清单
- 在 TUI 中查看接口详情并直接发起请求
- 在 macOS 与 Linux 上稳定运行

## 3. 产品目标

### 3.1 核心目标

V1 聚焦解决以下三个问题：

1. **找得到**：能够从 Spring MVC / Boot 项目中稳定识别 Controller 和接口方法
2. **看得懂**：能够展示接口路径、HTTP 方法、参数、请求体、来源文件等核心内容
3. **发得出**：能够在 TUI 中编辑参数并发送请求，查看响应结果

### 3.2 成功标准

- 对标准 Spring Boot 项目中的 Controller 扫描成功率达到可用水平
- 用户能够在 1 分钟内完成：扫描项目 -> 搜索接口 -> 编辑参数 -> 发起请求
- 在没有 Swagger 的情况下，仍能作为本地调试入口使用
- 在 macOS 与 Linux 上完成安装、扫描、请求发送的闭环

## 4. 产品定位

API TUI 不是在线 API 管理平台，也不是网关系统，而是：

- 面向开发者的本地终端工具
- 面向源码真实定义的接口发现与调试工具
- 面向“无文档 / 文档过期 / 需要快速联调”的本地工作流产品

## 5. 目标用户

### 5.1 主要用户

- Java 后端开发
- 需要联调接口的前端或全栈开发
- 测试开发 / 自动化测试工程师
- 偏好命令行与 TUI 工作流的工程师

### 5.2 用户特征

- 熟悉 Spring Boot 项目结构
- 需要快速定位接口定义
- 频繁在本地 / 测试环境验证接口
- 不希望频繁切换浏览器、Postman、IDE 与终端
- 使用 macOS 或 Linux 作为主要开发环境

## 6. 使用场景

### 场景 1：接手新项目

用户拿到一个 Spring Boot 项目，没有完整接口文档，希望快速了解有哪些可用接口。

### 场景 2：联调某个接口

用户知道业务模块，但不知道具体路径和参数定义，希望通过工具快速搜索接口并发起请求。

### 场景 3：没有 Swagger 或 Swagger 不可信

项目中已有注解定义，但文档页面缺失或过期，用户需要以源码为准进行调试。

### 场景 4：终端工作流

用户长期在终端内工作，希望使用键盘完成 API 浏览、编辑和发送请求。

### 场景 5：本地多环境验证

用户希望通过切换本地 dev / test 环境变量，在同一个接口模板上重复发起请求。

## 7. 产品范围

### 7.1 V1 包含范围

1. 扫描指定 Java 项目目录
2. 识别 Spring MVC / Boot 的 Controller 接口
3. 构建统一的本地 API 模型
4. 在 TUI 中展示接口列表与详情
5. 支持接口搜索与基础过滤
6. 支持编辑 Path / Query / Header / Body 参数
7. 支持发送 HTTP 请求并查看响应
8. 支持基础环境变量配置
9. 支持 macOS 与 Linux 系统安装运行

### 7.2 V1 不包含范围

1. FeignClient 扫描
2. JAX-RS 扫描
3. GraphQL / gRPC 支持
4. 自动生成完整 OpenAPI 文档
5. Windows 适配
6. 团队共享空间、云端同步、多人协作
7. 自动鉴权脚本、复杂签名编排
8. 自动化回归测试编排

## 8. 产品原则

1. **源码优先**：以项目源码中的 Spring 注解为真实来源
2. **稳定优先**：第一版优先保证 Spring MVC / Boot 扫描稳定性，不盲目扩框架
3. **最短路径**：用户能以最少步骤完成接口查找与调试
4. **终端友好**：优先键盘操作与文本界面可读性；布局与快捷键形成可记忆的工作流
5. **跨平台一致**：macOS 与 Linux 体验一致，安装和使用路径尽量统一
6. **性能与可分析性**：扫描与索引优先满足大仓库可用性；静态分析实现可测试、可分层演进（必要时可引入独立 Java 扫描子进程作为补充，不作为 V1 默认路径）

## 9. 核心用户流程

### 9.1 主流程

1. 用户输入项目目录
2. 工具扫描 Spring MVC / Boot 接口
3. 工具展示接口列表
4. 用户搜索并选中某个接口
5. 工具展示接口详情与默认参数结构
6. 用户编辑请求参数
7. 用户选择环境并发送请求
8. 工具展示响应结果

### 9.2 辅助流程

- 刷新扫描结果
- 按模块或 HTTP 方法过滤接口
- 查看接口来源文件与源码位置
- 保存最近使用的环境配置

## 10. 功能概览

### 模块 A：项目扫描
- 输入项目根目录
- 自动识别源码目录
- 扫描 Spring Controller 与映射关系
- 输出统一 API 数据模型

### 模块 B：接口浏览
- 列表展示全部接口
- 支持按关键字搜索
- 支持按 HTTP 方法过滤
- 支持查看接口详情

### 模块 C：请求编辑与发送
- 支持编辑 URL、Path、Query、Header、Body
- 支持基础环境变量替换
- 支持发送请求
- 支持查看响应状态、响应头、响应体、耗时

### 模块 D：本地配置
- 支持保存本地环境配置
- 支持保存工具基础偏好设置

## 11. 版本策略

### V1
聚焦 Spring MVC / Boot 扫描的稳定性与基础调试闭环；TUI 采用 Rust 生态（与 yazi 同栈叙事：ratatui + crossterm），交付单二进制或 `cargo install` 可安装产物。

### V1.1 候选增强：接口文件变动监听

在 V1 稳定可用的前提下，可支持「接口文件变动监听实时调整」：

- 监听项目源码目录中的 `.java` 文件变化（见 FR-14）
- 新增、修改、删除时触发接口索引更新；TUI 列表与详情自动刷新
- 以增量更新为主，异常时回退全量重扫
- 产品原则：防抖 + 增量优先；监听不可靠时优先保证索引一致性而非速度

### V2 候选方向
- Feign / JAX-RS 扫描
- OpenAPI 导出
- 历史请求与收藏
- 鉴权模板与前置脚本
- 增量扫描缓存

# 第二部分：需求文档（SRD）

## 12. 总体需求说明

系统输入为一个 Java 项目目录，**实现为 Rust 应用程序**；对 Spring MVC / Boot 相关源码进行**静态扫描**（基于 tree-sitter 等可测试的解析管线），提取本地 HTTP API 清单，并通过 **全屏 TUI**（多面板、键盘优先）提供浏览、编辑、请求发送与结果查看能力。**运行本工具不要求用户本机安装 Java**；被扫描对象仍为用户的 Java/Spring 项目源码。

## 13. 术语定义

- **项目目录**：用户提供的 Java 项目根路径
- **接口**：由 Spring MVC / Boot 注解定义的 HTTP Endpoint
- **本地 API 模型**：工具内部统一表示接口定义的数据结构
- **环境变量**：例如 `host`、`token` 等用于请求拼装的本地配置项
- **TUI**：Text-based User Interface，文本终端交互界面

## 14. 功能需求

## FR-01 项目目录输入

### 描述
系统应允许用户指定一个本地 Java 项目目录作为扫描输入。

### 约束
- 支持绝对路径与相对路径
- 路径不存在时给出明确错误提示
- 路径可通过命令参数传入

### 验收标准
- 用户输入有效目录后，系统能进入扫描流程
- 用户输入无效目录后，系统返回错误，不进入 TUI 主界面

## FR-02 Spring MVC / Boot 接口扫描

### 描述
系统应扫描项目中的 Spring MVC / Boot Controller，并识别接口方法。

### 必须识别的类注解
- `@RestController`
- `@Controller`
- `@RequestMapping`

### 必须识别的方法注解
- `@GetMapping`
- `@PostMapping`
- `@PutMapping`
- `@DeleteMapping`
- `@PatchMapping`
- `@RequestMapping`

### 识别内容
- Controller 类名
- 方法名
- HTTP Method
- 完整 Path
- 来源文件路径
- 所在类与方法位置

### 验收标准
- 标准 Spring Boot Controller 能被正确识别
- 类级路径与方法级路径能正确拼接
- `@RequestMapping(method=...)` 能被识别为具体 HTTP 方法

## FR-03 参数识别

### 描述
系统应识别接口参数，并按来源分类。

### 必须支持的参数注解
- `@PathVariable`
- `@RequestParam`
- `@RequestHeader`
- `@CookieValue`
- `@RequestBody`
- `@ModelAttribute`

### 需提取的信息
- 参数名
- Java 类型
- 参数位置
- 是否必填
- 默认值（如存在）

### 验收标准
- Path 参数能出现在路径占位变量列表中
- Query 参数能展示为可编辑键值项
- Body 参数能识别为请求体结构入口

## FR-04 请求体结构生成

### 描述
当接口包含 `@RequestBody` 参数时，系统应生成基础请求体模板。

### 最低要求
- 支持基础类型
- 支持普通 Java Bean 展开
- 支持简单集合类型
- 防止循环引用导致无限展开

### 验收标准
- 用户打开包含 Body 的接口时，可看到可编辑的默认 JSON 模板
- 复杂对象至少能生成结构化占位内容

## FR-05 本地 API 模型构建

### 描述
系统应将扫描结果归一化为统一的本地 API 模型，供 TUI 与请求执行模块使用。

### 模型至少包含字段
- 接口唯一 ID
- 接口名称
- 模块/控制器名
- HTTP Method
- Path
- 参数列表
- Body 模板
- 来源文件
- 行号或可定位信息

### 验收标准
- 不同 Controller 扫描出的接口可统一在一个列表中展示
- TUI 与请求执行不直接依赖源码 AST 结构

## FR-06 TUI 接口列表展示

### 描述
系统应在 TUI 中展示接口列表。

### 展示内容
- HTTP Method
- Path
- Controller/模块名
- 接口摘要信息

### 交互要求
- 支持上下移动选中
- 支持进入详情
- 支持返回列表
- 采用**全屏多面板**布局（如列表区 / 详情区 / 请求与响应区可分区展示），快捷键可发现（如帮助页）

### 验收标准
- 用户可从列表中浏览全部扫描到的接口
- 终端窗口缩放后界面仍可基本使用

## FR-07 搜索与过滤

### 描述
系统应支持对接口列表进行搜索与基础过滤。

### 支持的过滤条件
- 关键字搜索（Path、方法名、Controller 名）
- HTTP Method 过滤

### 验收标准
- 用户输入关键字后，列表结果实时缩小或刷新
- 用户选择 GET / POST 等过滤条件后，仅显示匹配接口

## FR-08 接口详情展示

### 描述
系统应展示选中接口的详细信息。

### 必须展示内容
- HTTP Method
- 完整 Path
- 来源 Controller 与方法名
- 来源文件路径
- Path / Query / Header / Body 参数
- 默认请求体模板

### 验收标准
- 用户可以不查看源码，也能理解接口的基本输入结构

## FR-09 请求编辑

### 描述
系统应允许用户编辑请求参数。

### 编辑范围
- Base URL
- Path 参数
- Query 参数
- Header
- Body JSON

### 验收标准
- 用户修改参数后，系统可据此生成最终请求
- Body 编辑错误时，系统能给出提示

## FR-10 请求发送

### 描述
系统应能够根据用户编辑内容发起 HTTP 请求。

### 最低支持
- GET
- POST
- PUT
- DELETE
- PATCH

### 验收标准
- 用户可在 TUI 内直接发起请求
- 请求发送失败时能看到错误信息

## FR-11 响应展示

### 描述
系统应展示请求结果。

### 必须展示内容
- HTTP 状态码
- 响应耗时
- 响应头
- 响应体文本

### 建议展示
- JSON Pretty Print
- 错误信息单独高亮

### 验收标准
- 用户能区分请求成功与失败
- JSON 响应具备基本可读性

## FR-12 环境配置

### 描述
系统应支持本地环境配置，以便在不同地址间切换请求目标。

### V1 最低支持
- `host`
- 常用 Header，例如 `Authorization`

### 验收标准
- 用户可切换 dev / test 等环境
- 环境变量变更后，最终请求地址与 Header 同步变化

## FR-13 刷新扫描

### 描述
系统应支持重新扫描当前项目。

### 验收标准
- 用户修改源码后，可重新触发扫描并更新接口列表

## FR-14 接口文件变动监听（增强需求）

### 描述
系统可支持监听项目源码目录中的接口相关文件变化，并自动调整本地 API 索引。

### 监听范围
- `src/main/java` 及其子目录
- 仅关注 `.java` 文件

### 支持事件
- 文件新增
- 文件修改
- 文件删除

### 系统行为
- 文件变化后经过短暂防抖再处理
- 优先对单文件执行增量重解析
- 新增文件时尝试加入监听范围
- 删除文件时移除该文件对应接口
- 当监听事件丢失、解析异常或索引状态不一致时，自动回退为全量重扫

### 验收标准
- 修改接口定义文件后，接口列表可自动更新
- 删除接口文件后，相关接口可自动从列表中移除
- 新增 Controller 文件后，相关接口可自动出现在列表中
- 自动刷新失败时，系统能够提示并执行降级处理

## FR-15 TUI 自动刷新（增强需求）

### 描述
当监听到接口索引变化时，TUI 应自动刷新展示结果。

### 系统行为
- 列表数据自动更新
- 当前选中项尽量保持稳定
- 若当前详情页对应接口已失效，给出提示并返回安全状态

### 验收标准
- 接口列表刷新时不发生界面崩溃
- 用户在大多数场景下无需手动刷新即可看到最新接口结果

## 15. 非功能需求

## NFR-01 操作系统支持

### 要求
V1 必须支持以下系统：
- macOS
- Linux

### 说明
- 两个平台均应支持安装、扫描项目、进入 TUI、发起请求
- 不要求 V1 支持 Windows

### 验收标准
- 在 macOS 上完成完整主流程验证
- 在 Linux 上完成完整主流程验证
- 不依赖仅某一操作系统可用的 GUI 能力

## NFR-02 安装与运行方式

### 要求
工具应提供统一、清晰的本地运行方式；**默认分发形态为 Rust 原生可执行文件**（或源码 `cargo install` / `cargo run`），**不依赖用户安装 JVM**。

### 建议方案
- 发布 **静态或动态链接的单一二进制**（按平台提供 release 产物）
- 支持从源码 **`cargo build --release`** 构建
- 支持通过子命令或参数指定项目目录并进入 TUI（详见开发文档）

### 验收标准
- 用户无需安装 Java、无需 IDE 集成即可运行
- macOS 与 Linux 启动方式一致或基本一致（同一 CLI 界面）

## NFR-03 性能

### 要求
在中小型 Spring Boot 项目中，首次扫描时间需控制在可接受范围。

### 目标
- 中小项目扫描在数秒级内完成
- TUI 基础操作响应流畅
- 启用自动监听时，文件变更后的接口刷新应在可接受时间内完成

### 验收标准
- 扫描过程中有明确状态提示
- 列表滚动与详情切换无明显卡顿
- 文件修改后，接口视图能在短时间内完成刷新

## NFR-04 稳定性

### 要求
对常见 Spring MVC 写法应具备稳定识别能力，对无法识别情况应可降级。

### 验收标准
- 遇到单个文件解析失败，不应导致整体崩溃
- 失败文件应记录并提示
- 自动监听场景下，事件异常或索引不一致时可自动回退到全量重扫

## NFR-05 易用性

### 要求
TUI 应尽量以键盘为主完成主要操作；布局与交互参考 **yazi / lazygit** 类工具：多面板、可发现快捷键（如 `?` 帮助）、状态栏提示。

### 验收标准
- 支持基础快捷键导航与列表/详情切换
- 关键操作具备状态提示
- 错误信息可理解

## NFR-06 可扩展性

### 要求
架构上需为后续扩展其他框架扫描器预留空间；Rust 侧以 **crate 模块**划分扫描、模型、TUI、HTTP，便于单测与替换实现。

### 验收标准
- 扫描层与 TUI、请求执行层解耦
- Spring 扫描逻辑可作为独立模块（crate 内 `scanner` 等）存在与测试
- 若未来引入 **Java 侧辅助扫描进程**，接口边界为 **JSON 交换**，不侵入 TUI 主循环

## 16. 兼容性要求

### Java 项目兼容范围
- Maven 项目
- Gradle 项目
- 标准 Spring Boot 项目结构优先

### Spring 使用范围
V1 仅保证对常见 Spring MVC / Boot 注解组合具备较高兼容性，不要求覆盖所有极端自定义写法。

## 17. 技术约束

1. V1 采用源码静态扫描为主；实现语言为 **Rust**，解析管线以 **tree-sitter-java**（及配套查询）为基线
2. 不依赖项目必须先成功编译；**不要求用户本机安装 Java** 以运行本工具
3. TUI 基于 **ratatui + crossterm**，需兼容 macOS 与 Linux 终端（含常见终端尺寸变化）
4. HTTP 客户端使用 Rust 生态（如 **reqwest** 等），与 TUI 线程模型协调（避免阻塞渲染）
5. 本地配置存储方式需兼容 macOS 与 Linux 文件系统
6. 不强依赖桌面 GUI、浏览器或系统特定能力
7. 可选后续：若需覆盖极端 Spring 语义，可通过 **独立 Java 扫描子进程 + JSON** 扩展，不作为 V1 默认依赖

## 19. 异常与边界处理

### 异常场景
- 项目目录无效
- 未找到 Spring Controller
- 某源码文件解析失败
- 参数结构无法完全推断
- 请求发送失败
- 响应体非 JSON
- 文件监听事件丢失或重复触发
- 自动刷新过程中当前选中接口已失效

### 系统处理原则
- 给出可理解的提示
- 尽量保留已成功扫描的结果
- 不因局部异常导致整体不可用
- 监听异常时优先保证接口索引一致性

## 22. V1 结论

V1 的定位应明确收敛为：

**一个支持 macOS 与 Linux 的、本地运行的 Spring MVC / Boot 接口扫描与调试 TUI 工具；由 Rust 实现，终端体验对齐 yazi / lazygit 所代表的现代全屏 TUI 水准，用户无需安装 Java 即可运行本工具。**

第一版不追求“全框架覆盖”，而追求“Spring 场景稳定、流程完整、终端体验顺手”。只有先把 Spring MVC / Boot 做稳，后续扩展 Feign、JAX-RS、OpenAPI 导出、历史记录与自动化测试才有意义。
